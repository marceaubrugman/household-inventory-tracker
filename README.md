# Household Inventory Tracker

A Python and PostgreSQL household inventory application with both a console interface and a FastAPI REST API, built with a layered backend architecture.

**Current release: v0.3.0**

## Overview

The Household Inventory Tracker, or HIT, helps users record, retrieve, search, update, sort, delete, and monitor household items.

The project is being developed incrementally as a practical Python data and backend engineering portfolio project:

* **v0.1.0:** Python console application with JSON persistence
* **v0.2.0:** PostgreSQL became the runtime source of truth
* **v0.3.0:** FastAPI added a tested HTTP interface while preserving the console application

HIT now demonstrates two interfaces over the same PostgreSQL-backed data-access layer:

* a menu-driven console application
* a documented REST API with validated CRUD endpoints

## Features

### Console application

* Add household inventory items
* View all stored items
* Search by name, category, or location
* Update existing items
* Delete items with confirmation
* Sort by:

  * name
  * category
  * location
  * quantity
* View items at or below their minimum quantity
* Handle unavailable database connections gracefully

### FastAPI REST API

* Check API liveness
* List all inventory items
* Retrieve one item by ID
* Create inventory items
* Partially update inventory items
* Delete inventory items
* Validate request bodies and path parameters
* Return controlled `404`, `422`, and `503` responses
* Generate interactive Swagger and OpenAPI documentation automatically

### Data and migration

* Store inventory persistently in PostgreSQL
* Generate item IDs in PostgreSQL
* Enforce database constraints
* Migrate legacy JSON inventory data to PostgreSQL
* Run isolated PostgreSQL integration tests against `hit_test`

## Item Structure

Each inventory item contains:

```python
{
    "id": 1,
    "name": "Rice",
    "category": "Food",
    "location": "Pantry",
    "quantity": 2,
    "minimum_quantity": 3,
    "notes": ""
}
```

The item ID is generated by PostgreSQL.

## Architecture

HIT v0.3.0 provides two interfaces over shared repository and database layers.

### Console path

```text
app.py
   ↓
inventory_workflows.py
   ↓
item_repository.py
   ↓
database.py
   ↓
PostgreSQL
```

### API path

```text
Uvicorn
   ↓
FastAPI
   ↓
API routers and dependencies
   ↓
item_service.py
   ↓
item_repository.py
   ↓
database.py
   ↓
PostgreSQL
```

FastAPI does not connect directly to PostgreSQL. API requests pass through Python application, repository, and database layers. SQL remains isolated inside `item_repository.py`.

The console application currently retains its existing workflow layer, while the API uses `item_service.py` as an application-service boundary.

### Layer responsibilities

* `app.py`

  * runs the console application loop
  * routes menu choices
  * handles expected database failures

* `inventory_workflows.py`

  * collects console input
  * applies validation and normalization
  * coordinates repository operations
  * displays results

* `src/api/main.py`

  * creates the FastAPI application
  * registers exception handlers
  * includes API routers

* `src/api/routers/items.py`

  * defines inventory HTTP endpoints
  * maps application results to HTTP status codes
  * returns validated API responses

* `src/api/dependencies.py`

  * connects FastAPI dependency injection to application operations
  * validates item IDs used by read endpoints

* `src/api/schemas.py`

  * defines Pydantic request and response models
  * validates and normalizes API input
  * controls the public response shape

* `src/api/exception_handlers.py`

  * converts known database failures into safe `503` responses
  * logs PostgreSQL operational failures without exposing internal details to clients

* `item_service.py`

  * provides interface-independent application operations for the API
  * coordinates repository calls
  * merges partial updates
  * prevents unsupported fields from being updated

* `item_repository.py`

  * contains PostgreSQL queries
  * performs create, read, update, and delete operations
  * provides search, sorting, and low-stock retrieval
  * keeps SQL out of interface and service code

* `database.py`

  * reads database configuration
  * opens PostgreSQL connections
  * applies a connection timeout

* PostgreSQL

  * stores inventory records
  * generates item IDs
  * enforces data constraints

## API Endpoints

| Method | Path | Purpose | Success status |
|---|---|---|---:|
| `GET` | `/health` | Check API liveness | `200` |
| `GET` | `/items` | List all inventory items | `200` |
| `GET` | `/items/{item_id}` | Retrieve one inventory item | `200` |
| `POST` | `/items` | Create an inventory item | `201` |
| `PATCH` | `/items/{item_id}` | Partially update an inventory item | `200` |
| `DELETE` | `/items/{item_id}` | Delete an inventory item | `204` |

### API behavior

| Situation | Response |
|---|---:|
| Successful read or update | `200` |
| Successful creation | `201` |
| Successful deletion | `204` |
| Item not found | `404` |
| Invalid path parameter or request body | `422` |
| Database configuration missing | `503` |
| PostgreSQL operationally unavailable | `503` |

A successful deletion returns an empty response body.

### Example creation request

```json
{
  "name": "Rice",
  "category": "Food",
  "quantity": 3,
  "minimum_quantity": 1,
  "location": "Pantry",
  "notes": "Basmati"
}
```

The backend controls generated fields such as the item ID.

### Example partial update

```json
{
  "quantity": 8,
  "notes": "Updated through the API"
}
```

Only supplied fields are changed. Omitted fields retain their existing values.

## Security and Reliability

HIT v0.3.0 includes several defensive programming practices:

* parameterized Psycopg queries
* no direct insertion of user values into SQL strings
* allowlisted SQL sort expressions
* allowlisted service-layer update fields
* Pydantic request and response validation
* positive-integer validation for item IDs
* database-level quantity constraints
* credentials stored outside source code
* connection timeout handling
* controlled `404`, `422`, and `503` API responses
* safe public database-error messages
* server-side logging of PostgreSQL operational failures
* isolated PostgreSQL integration-test database
* guarded JSON migration
* atomic migration transactions

Application validation improves the user experience, while PostgreSQL constraints provide a final data-integrity boundary.

## Installation

Clone the repository:

```bash
git clone https://github.com/marceaubrugman/household-inventory-tracker.git
cd household-inventory-tracker
```

Create a virtual environment:

```bash
python -m venv .venv
```

Activate it on Windows PowerShell:

```powershell
.venv\Scripts\Activate.ps1
```

Activate it on macOS or Linux:

```bash
source .venv/bin/activate
```

Install the dependencies:

```bash
python -m pip install -r requirements.txt
```

## PostgreSQL Setup

### 1. Create the application database

Create a PostgreSQL database named:

```text
hit_db
```

### 2. Apply the schema

Open `sql/schema.sql` in pgAdmin or another PostgreSQL client and execute it inside `hit_db`.

The script creates:

```text
hit.items
```

### 3. Configure the database URL

HIT reads the normal application connection string from:

```text
DATABASE_URL
```

Example format:

```text
postgresql://USERNAME:PASSWORD@localhost:5432/hit_db
```

Do not store the real connection string in source code or commit it to Git.

In PyCharm, add `DATABASE_URL` to the relevant console and Uvicorn Run/Debug Configurations.

For a temporary PowerShell session:

```powershell
$env:DATABASE_URL = "postgresql://USERNAME:PASSWORD@localhost:5432/hit_db"
```

For macOS or Linux:

```bash
export DATABASE_URL="postgresql://USERNAME:PASSWORD@localhost:5432/hit_db"
```

## Running the Console Application

Make sure PostgreSQL is running and `DATABASE_URL` is configured, then execute:

```bash
python app.py
```

The menu provides access to the console inventory operations.

## Running the FastAPI Application

Make sure PostgreSQL is running and `DATABASE_URL` is configured, then start Uvicorn from the project root:

```bash
python -m uvicorn src.api.main:app --reload
```

The API is available at:



OpenAPI schema:


## Testing

HIT includes unit tests, API endpoint tests, service tests, migration tests, and PostgreSQL integration tests.

### Tests that do not require PostgreSQL

Run:

```bash
python -m pytest -m "not integration" -v
```

These tests use fakes, dependency overrides, and monkeypatching to isolate the API and service layers from PostgreSQL.

### Integration tests

Create a separate PostgreSQL database named:

```text
hit_test
```

Apply `sql/schema.sql` inside `hit_test`.

Configure:

```text
TEST_DATABASE_URL
```

Example format:

```text
postgresql://USERNAME:PASSWORD@localhost:5432/hit_test
```

Then run:

```bash
python -m pytest -m integration -v
```

The integration-test fixture:

* redirects repository calls to `hit_test`
* cleans the table before and after each test
* resets generated IDs
* refuses to clean databases whose names do not end with `_test`

The normal development database and isolated test database must remain separate.

### Complete test suite

With `TEST_DATABASE_URL` configured:

```bash
python -m pytest -ra
```

When `TEST_DATABASE_URL` is not configured, PostgreSQL integration tests are skipped intentionally and the remaining tests still run.

### Run all API tests in PowerShell

```powershell
python -m pytest (Get-ChildItem tests\test_api_*.py).FullName -v
```

PowerShell does not expand `tests/test_api_*.py` in the same way as Bash, so the `Get-ChildItem` command resolves the filenames first.

## JSON-to-PostgreSQL Migration

HIT includes a migration tool for importing inventory data from the v0.1 JSON format.

### Validate without writing data

```bash
python -m scripts.migrate_json_to_postgres sample_inventory.json --dry-run
```

### Run a migration

```bash
python -m scripts.migrate_json_to_postgres inventory.json
```

The migration tool:

* validates every record before connecting to PostgreSQL
* preserves existing item IDs
* rejects duplicate IDs
* rejects invalid quantities and missing fields
* requires an empty target table
* imports all records in one transaction
* updates the PostgreSQL identity sequence

The script requires explicit confirmation before writing data.

Private household inventory files should not be committed to Git.

## Database Schema

The initial PostgreSQL model uses a single inventory table:

```sql
CREATE TABLE IF NOT EXISTS hit.items (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    category VARCHAR(100) NOT NULL,
    location VARCHAR(100) NOT NULL,
    quantity INTEGER NOT NULL
        CONSTRAINT quantity_non_negative CHECK (quantity >= 0),
    minimum_quantity INTEGER NOT NULL
        CONSTRAINT minimum_quantity_non_negative
        CHECK (minimum_quantity >= 0),
    notes TEXT NOT NULL DEFAULT ''
);
```

A single table is intentionally used at this stage to keep the model understandable while establishing a reliable PostgreSQL and API foundation.
